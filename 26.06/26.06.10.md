# TIL - JPA 시작: 트랜잭션, 영속성 컨텍스트, 엔티티 상태, 매핑

## 1. 학습 주제

이번에는 JPA를 본격적으로 시작하면서, 왜 JPA가 필요한지부터 `EntityManager`, 영속성 컨텍스트, 엔티티 상태 변화, 매핑, 그리고 `@Transactional`까지 한 흐름으로 정리하게 되었다.  
코드만 보면 각각 따로 이해되는 것 같지만, 실제로는 모두 연결되어 있다.

가장 큰 흐름은 이렇다.

- JDBC나 MyBatis는 SQL 중심이다.
- JPA는 객체 중심으로 DB를 다루려는 기술이다.
- 그 객체를 관리하는 핵심 장치가 영속성 컨텍스트이다.
- 트랜잭션은 그 관리된 변경 내용을 DB에 안전하게 반영하기 위한 단위이다.

즉 JPA를 이해하려면 “엔티티를 조회하고 저장하는 문법”보다  
**누가 객체를 관리하고, 언제 DB에 반영되며, 왜 트랜잭션이 필요한지**를 먼저 잡아야 한다.

---

## 2. JPA, ORM, Hibernate 정리

### JPA
JPA는 자바 ORM 표준 인터페이스이다.  
정확히 말하면 “자바에서 객체와 데이터베이스를 어떻게 연결해서 다룰지에 대한 표준”이다.

즉 JPA 자체가 바로 동작하는 엔진이라기보다,  
“이런 방식으로 동작해야 한다”는 규약에 가깝다.

### ORM
ORM은 Object Relational Mapping의 줄임말이다.  
쉽게 말하면 자바 객체와 관계형 데이터베이스 테이블을 연결해주는 방식이다.

예를 들어:

- 자바의 `Member` 객체
- DB의 `tbl_member` 테이블

을 서로 매핑해서,  
객체를 저장하면 테이블에 INSERT가 일어나고,  
테이블에서 조회하면 객체로 바꿔주는 개념이다.

### Hibernate
Hibernate는 JPA 표준을 실제로 구현한 대표 구현체이다.

즉 관계를 정리하면:

- JPA = 표준
- Hibernate = 구현체

이다.

---

## 3. 왜 JPA가 나왔는가

JDBC로 직접 SQL을 다루거나, MyBatis처럼 SQL을 직접 작성하는 방식은 장점도 많지만 한계도 있다.

대표적으로 다음 문제가 생긴다.

### 1. 중복 코드가 많다
조회할 때마다:

- SQL 작성
- Statement/PreparedStatement 생성
- ResultSet 읽기
- 객체에 값 세팅

을 반복해야 한다.

### 2. SQL 의존성이 강하다
조회 컬럼이 조금만 바뀌어도 Java 코드도 같이 수정해야 한다.

### 3. 객체지향과 DB 사이에 거리감이 있다
객체는:

- 상속
- 연관관계
- 참조

를 가지는데, DB는:

- 테이블
- 컬럼
- 외래키

중심이다.

즉 객체지향 모델과 관계형 모델이 완전히 같지 않다.

### 4. 동일성 보장이 어렵다
같은 행을 두 번 조회해도 Java에서는 서로 다른 객체가 된다.

JPA는 이런 문제를 줄이기 위해 등장했다.  
즉 JPA는 단순히 SQL을 덜 쓰는 기술이 아니라,  
**객체 중심으로 데이터를 다루기 위한 기술**이다.

---

## 4. 엔티티란 무엇인가

JPA에서 가장 중요한 단어 중 하나가 엔티티이다.

### 엔티티
엔티티는 **DB 테이블과 매핑되는 JPA 관리 객체**이다.

여기서 중요한 점은, 엔티티는 단순 DTO와 다르다는 것이다.

- DTO는 보통 데이터 전달용 객체
- 엔티티는 JPA가 상태를 추적하고 관리하는 객체

즉 엔티티는:

- 테이블과 연결되어 있고
- 기본 키를 가지고 있고
- JPA가 생명주기를 관리하는 객체

라고 이해하는 것이 정확하다.

그래서 엔티티는 그냥 값 보관 상자가 아니라,  
**JPA가 “DB에 반영할 대상”으로 인식하는 객체**이다.

---

## 5. 매핑이란 무엇인가

“매핑이 뭐지?”가 JPA 초반에 가장 헷갈린다.

### 매핑
매핑은 **자바 객체의 필드와 DB 테이블의 컬럼을 연결하는 것**이다.

예를 들어:

- 클래스 이름 -> 테이블 이름
- 필드 이름 -> 컬럼 이름
- enum 값 -> 문자열 또는 숫자
- 저장하지 않을 필드 -> 제외

같은 규칙을 정하는 것이다.

즉 매핑은 “JPA가 이 객체를 DB에 어떻게 저장하고 읽을지 알려주는 설정”이다.

이번에 본 대표적인 매핑은 다음과 같다.

### `@Entity`
이 클래스는 엔티티라는 뜻이다.

### `@Table`
이 엔티티가 어떤 테이블과 연결되는지 지정한다.

### `@Id`
기본 키를 뜻한다.

### `@GeneratedValue`
기본 키 값을 DB가 자동 생성하도록 할 때 사용한다.

### `@Column`
필드가 어떤 컬럼과 연결되는지, 길이/nullable/unique 같은 조건을 지정할 수 있다.

### `@Enumerated`
enum을 DB에 어떻게 저장할지 정한다.

### `@Transient`
이 필드는 DB에 저장하지 않는다는 뜻이다.

즉 매핑은  
**객체 구조를 데이터베이스 구조에 맞게 연결하는 번역 규칙**이라고 생각하면 된다.

---

## 6. `@Transient`, `@Enumerated`가 의미하는 것

이번 매핑 예제에서 특히 눈에 띄는 부분은 이 두 개였다.

### `@Transient`
해당 필드는 엔티티 안에는 있지만 DB 컬럼으로 저장하지 않는다.

즉 자바에서는 사용하지만 DB에는 반영되지 않는 값이다.

### `@Enumerated(EnumType.STRING)`
enum을 DB에 문자열로 저장한다.

예를 들어:

- `ROLE_MEMBER`
- `ROLE_ADMIN`

이런 값을 그대로 문자열로 저장한다.

이런 설정이 중요한 이유는,  
JPA는 객체를 무조건 통째로 저장하는 게 아니라  
**어떤 필드는 저장하고, 어떤 방식으로 저장할지**도 세밀하게 정하기 때문이다.

---

## 7. `EntityManagerFactory`와 `EntityManager`

이 둘도 초반에 매우 헷갈리지만 역할을 나누면 생각보다 단순하다.

### `EntityManagerFactory`
`EntityManager`를 만들어내는 공장이다.  
무겁고 비싼 객체라 보통 애플리케이션 전체에서 하나만 만들어 재사용한다.

즉 개념적으로는 singleton처럼 생각하면 된다.

### `EntityManager`
엔티티를 실제로 다루는 관리자이다.

이걸 통해:

- 조회
- 저장
- 삭제
- 상태 관리

를 수행한다.

정리하면:

- `EntityManagerFactory` = 공장
- `EntityManager` = 실무 담당자

이다.


- `EntityManagerFactory`: singleton 감각
- `EntityManager`: 보통 요청이나 작업 단위에서 사용

즉 `EntityManager`는 계속 새로 만들어 써도 되지만,  
그걸 만드는 공장인 `EntityManagerFactory`는 매번 새로 만들면 비효율적이다.

---

## 8. 영속성 컨텍스트란 무엇인가

이 부분이 JPA의 진짜 핵심이다.

### 영속성 컨텍스트
영속성 컨텍스트는 **엔티티를 관리하는 메모리 공간**이다.

조금 더 풀면:

- 엔티티를 읽거나 저장하면
- JPA가 그 객체를 그냥 버리지 않고
- “관리 중인 객체”로 메모리 안에 보관한다

이 관리 공간이 바로 영속성 컨텍스트이다.

즉 JPA는 단순히 SQL을 날리고 끝나는 게 아니라,  
**엔티티를 메모리에 올려놓고 상태를 계속 추적한다**는 점이 핵심이다.

---

## 9. 영속성 컨텍스트가 하는 일

영속성 컨텍스트는 단순 보관함이 아니다.  
주요 역할은 다음과 같다.

### 1. 엔티티 관리
현재 어떤 엔티티가 관리 중인지 기억한다.

### 2. 동일성 보장
같은 영속성 컨텍스트 안에서 같은 PK의 엔티티를 다시 조회하면 같은 객체를 반환할 수 있다.

### 3. 변경 감지
관리 중인 엔티티의 필드 값이 바뀌면, 나중에 DB 반영 시점에 그 변경을 감지할 수 있다.

### 4. SQL 실행 시점 조절
바로바로 SQL을 날리는 것이 아니라, 적절한 시점에 모아서 반영할 수 있다.

즉 영속성 컨텍스트는  
**엔티티 상태 추적 시스템**이라고 보면 된다.

---

## 10. 엔티티 상태 4가지

1. 비영속
2. 영속
3. 준영속
4. 삭제

이걸 정확히 구분하는 것이 중요하다.

### 1. 비영속
JPA가 전혀 관리하지 않는 객체 상태이다.

예를 들어:

```java
new Menu(...)
```

로 만든 객체는 그냥 평범한 Java 객체일 뿐이다.  
아직 DB와도 연결되지 않았고, 영속성 컨텍스트도 모른다.

즉 “메모리에만 있는 일반 객체”라고 보면 된다.

### 2. 영속
영속성 컨텍스트가 관리하는 상태이다.

예를 들어:

- `find()`로 조회한 엔티티
- `persist()`로 저장 대상으로 등록한 엔티티

는 영속 상태가 된다.

이 상태에서는 JPA가 객체를 추적한다.

### 3. 준영속
원래 영속 상태였는데, 더 이상 영속성 컨텍스트가 관리하지 않는 상태이다.

즉 한때는 관리 대상이었지만, 지금은 빠져나온 상태이다.

### 4. 삭제
삭제 대상으로 등록된 상태이다.

즉 엔티티를 제거 대상으로 표시해둔 상태라고 보면 된다.

---

## 11. 비영속과 영속의 차이

이 차이를 감각적으로 이해하는 게 가장 중요하다.

### 비영속
- 그냥 내가 만든 객체
- JPA가 모름
- 값 바꿔도 아무 일 안 일어남

### 영속
- JPA가 관리하는 객체
- 값 바꾸면 JPA가 추적 가능
- 트랜잭션 커밋 시 DB 반영 가능

즉 둘의 차이는  
**JPA가 그 객체를 알고 있느냐, 모르느냐**다.

---

## 12. `persist()`는 무엇인가

`persist()`도 아주 중요하다.

### `persist()`
비영속 객체를 영속성 컨텍스트에 등록해서 **영속 상태로 만드는 것**이다.

예를 들면:

```java
Menu newMenu = new Menu(...);
entityManager.persist(newMenu);
```

이 흐름은:

1. `newMenu`는 원래 비영속
2. `persist()`를 호출하면 영속 상태가 됨
3. 트랜잭션 커밋 시점에 INSERT가 반영될 수 있음

즉 `persist()`는 단순 “바로 저장”이 아니라,  
**저장 대상 엔티티로 등록하는 것**에 더 가깝다.

이 부분이 SQL 중심 사고와 가장 다른 지점이다.

- JDBC/MyBatis 감각: 지금 INSERT 실행
- JPA 감각: 이 엔티티를 저장 관리 대상으로 등록

이다.

---

## 13. 트랜잭션은 왜 필요한가

이제 트랜잭션을 JPA 관점으로 다시 보자.

### 트랜잭션
여러 작업을 하나의 작업 단위로 묶는 것

JPA에서는 특히 더 중요하다.  
왜냐하면 영속성 컨텍스트가 관리하는 변경 내용을 실제 DB에 반영하는 시점이  
대부분 **트랜잭션 커밋과 강하게 연결**되기 때문이다.

즉 트랜잭션이 필요한 이유는:

- 중간 실패 시 rollback 하기 위해서
- 데이터 일관성을 맞추기 위해서
- 변경 내용을 안전하게 확정하기 위해서

도 있지만,

JPA에서는 추가로:

- 영속성 컨텍스트의 변경 내용을
- 언제 DB에 반영할지 결정하는 기준

이 되기도 한다.

즉 트랜잭션은 단순히 “에러 나면 되돌리는 장치”를 넘어서,  
**JPA에서 변경 확정의 경계**라고 보면 된다.

---

## 14. JPA에서 수정은 왜 `update` 메서드 없이 되는가

이 부분이 처음엔 되게 신기하다.

코드에서 엔티티를 조회한 뒤:

```java
foundMenu.setMenuName(menuName);
```

만 하고 commit했다.

그런데도 DB가 바뀐다.

이유는:

1. `foundMenu`는 영속 상태다
2. 영속성 컨텍스트가 이 객체를 관리하고 있다
3. 필드 값이 바뀐 걸 JPA가 추적한다
4. commit 시점에 UPDATE SQL을 실행한다

즉 JPA는 직접 `update menu ...`를 쓰는 게 아니라  
**영속 상태 엔티티의 변경을 감지해서 반영**한다.

이걸 JPA의 변경 감지라고 이해하면 좋다.

---

## 15. `flush()`는 무엇인가

필기에는 없지만 상태 이해에 꼭 필요해서 같이 정리하면,

### `flush()`
영속성 컨텍스트의 변경 내용을 DB에 동기화하는 것

즉:

- INSERT
- UPDATE
- DELETE

예정 내용을 실제 SQL로 DB에 반영하는 작업이다.

다만 중요한 점은:

- `flush()`가 됐다고 무조건 최종 확정은 아니다
- 트랜잭션이 commit되지 않으면 rollback 가능하다

즉 `flush()`는 “DB에 보내기”,  
`commit()`은 “최종 확정”이라고 생각하면 된다.

---

## 16. 준영속 상태는 왜 필요한가

준영속은 처음엔 가장 헷갈린다.

### 준영속
영속 상태였던 엔티티를 영속성 컨텍스트가 더 이상 관리하지 않게 만든 상태

이 상태에서는:

- 객체 값은 바꿀 수 있음
- 하지만 JPA는 그 변경을 추적하지 않음

즉 “객체는 살아 있지만 JPA 관리 대상에서는 빠진 상태”다.

---

## 17. `detach()`는 무엇인가

### `detach()`
특정 엔티티 하나를 준영속 상태로 만든다.

예를 들어:

1. 엔티티 조회 -> 영속 상태
2. `detach(entity)` 호출
3. 이제 그 객체는 더 이상 JPA가 관리하지 않음

그래서 이후에 값 바꿔도 자동 반영되지 않는다.

즉 `detach()`는  
**특정 엔티티 하나만 관리 대상에서 빼는 것**이다.

---

## 18. `merge()`는 무엇인가

준영속 상태의 엔티티를 다시 관리 대상으로 합치고 싶을 때 쓴다.

### `merge()`
준영속 객체의 상태를 다시 영속성 컨텍스트에 반영하는 것

즉:

1. detach된 객체를 수정한다
2. `merge()`를 호출한다
3. JPA가 그 상태를 다시 관리 대상으로 받아들인다

그러면 이후 flush/commit 시 반영될 수 있다.

즉 `merge()`는  
**준영속 객체를 다시 영속성 관리 흐름으로 가져오는 작업**이다.

---

## 19. `clear()`와 `close()` 차이

이 둘도 같이 헷갈리기 쉽다.

### `clear()`
영속성 컨텍스트 안의 모든 엔티티를 준영속 상태로 만든다.

즉 관리 대상 목록을 비우는 것이다.

- `detach()`는 하나만
- `clear()`는 전부

라는 차이가 있다.

### `close()`
영속성 컨텍스트 자체를 종료한다.

즉 더 이상 그 `EntityManager`를 사용할 수 없다.

정리하면:

- `detach()` = 특정 엔티티만 관리 해제
- `clear()` = 전체 엔티티 관리 해제
- `close()` = `EntityManager` 자체 종료

이다.

---

## 20. `remove()`는 무엇인가

### `remove()`
영속 상태의 엔티티를 삭제 상태로 만든다.

중요한 점은 이것도 바로 DB에서 완전히 끝나는 감각보다:

- 삭제 대상으로 표시
- flush/commit 시 DELETE 반영

흐름으로 보는 것이 더 정확하다.

즉 `remove()`도 JPA답게  
“SQL을 바로 실행”보다  
“엔티티 상태를 삭제 상태로 바꿈”에 더 가깝다.

---

## 21. `given`, `when`, `then`은 무엇인가


### given
테스트를 위한 준비 단계

예:
- 새 메뉴 객체 만들기
- 회원 등록 DTO 만들기
- 테스트할 입력값 준비

### when
실제로 테스트 대상 동작을 수행하는 단계

예:
- `persist()` 호출
- `find()` 호출
- 수정 메서드 실행

### then
결과를 검증하는 단계

예:
- 값이 바뀌었는지
- 개수가 증가했는지
- 예외가 발생하지 않았는지

즉 `given / when / then`은 문법이 아니라  
**테스트를 읽기 쉽게 나누는 사고 방식**이다.

테스트를 읽을 때도:

- 무엇을 준비했고
- 무엇을 실행했고
- 무엇을 검증하는지

가 한눈에 보여서 좋다.

---

## 22. `@ParameterizedTest`와 소스 어노테이션들

반복해서 비슷한 테스트를 할 때 사용하는 방식이다.

### `@ParameterizedTest`
하나의 테스트 메서드를 여러 입력값으로 반복 실행하게 해준다.

즉 테스트 케이스를 여러 번 돌리고 싶을 때 유용하다.

---

## 23. `@ValueSource`

가장 단순한 값 목록을 넣을 때 쓴다.

예:
- 숫자 여러 개
- 문자열 여러 개

처럼 단일 값 하나씩 테스트할 때 적합하다.

즉 “간단한 입력 여러 개”를 줄 때 좋다.

---

## 24. `@CsvSource`

쉼표로 구분된 여러 값을 한 줄 단위로 넣을 때 사용한다.

예:
- `메뉴코드, 기대값`
- `id, 이름`

처럼 여러 파라미터를 동시에 넘길 수 있다.

즉 입력값이 둘 이상일 때 많이 쓴다.

---

## 25. `@MethodSource`

조금 더 복잡한 테스트 데이터를 메서드에서 만들어 공급할 때 쓴다.

예:
- 여러 필드를 가진 객체 조합
- 복잡한 시나리오 데이터
- 동적인 테스트 케이스

즉 `ValueSource`, `CsvSource`로 표현하기 어려운 경우에 적합하다.

정리하면:

- `@ValueSource` = 단일 값 여러 개
- `@CsvSource` = 여러 컬럼의 고정 값
- `@MethodSource` = 복잡한 테스트 데이터

이다.

---

## 26. `@PersistenceContext`는 무엇인가

이것도 JPA에서 아주 중요하다.

### `@PersistenceContext`
Spring이 관리하는 `EntityManager`를 주입받기 위한 어노테이션

즉 Repository에서:

- 내가 직접 `EntityManager`를 만들지 않고
- Spring/JPA가 관리하는 적절한 `EntityManager`를 넣어주는 방식이다

이 어노테이션 이름 자체가  
“영속성 컨텍스트와 연결된 `EntityManager`를 쓰겠다”는 느낌으로 이해해도 좋다.

즉 `EntityManager`는 결국 영속성 컨텍스트를 다루는 창구이고,  
`@PersistenceContext`는 그 창구를 스프링 환경에서 주입받는 방식이라고 보면 된다.

---

## 27. Spring에서 `@Transactional`과 JPA가 만나면

MyBatis에서와 비슷하게, JPA에서도 `@Transactional`은 Spring Proxy로 동작한다.

흐름은 이렇다.

1. Service 메서드 호출
2. Spring Proxy가 가로챔
3. 트랜잭션 시작
4. `EntityManager`와 영속성 컨텍스트가 그 트랜잭션 범위 안에서 동작
5. `persist`, 엔티티 수정, `remove` 등 작업 수행
6. commit 시 flush + 최종 반영
7. 예외 시 rollback

즉 JPA에서는 `@Transactional`이 더욱 중요하다.  
왜냐하면 영속성 컨텍스트에서 관리하던 엔티티 변경이  
대부분 **트랜잭션 종료 시점**에 최종 반영되기 때문이다.

---

## 28. 이번 범위 전체 흐름 정리

이번 공부를 한 줄로 정리하면:

- JPA는 객체를 DB에 매핑해서 관리하려는 기술이고
- 영속성 컨텍스트가 그 객체 상태를 관리하며
- `EntityManager`가 그 창구 역할을 하고
- 트랜잭션이 그 변경의 확정 경계가 된다

즉 순서는 이런 느낌이다.

1. 엔티티를 조회하거나 생성한다
2. 영속성 컨텍스트가 관리한다
3. 상태가 바뀌면 변경을 추적한다
4. 트랜잭션 커밋 시 DB에 반영한다

이 흐름이 이해되면,  
JPA의 `persist`, `find`, `remove`, `detach`, `merge`가 훨씬 자연스럽게 보이기 시작한다.

---

## 29. 핵심 정리

1. JPA는 자바 ORM 표준이고, Hibernate는 그 대표 구현체이다.
2. ORM은 자바 객체와 DB 테이블을 연결해주는 방식이다.
3. 엔티티는 JPA가 관리하는 객체이며, 단순 DTO와는 다르다.
4. 매핑은 객체 필드와 테이블/컬럼을 연결하는 규칙이다.
5. `EntityManagerFactory`는 `EntityManager`를 만드는 공장이고, 보통 하나만 만든다.
6. `EntityManager`는 엔티티를 조회, 저장, 삭제하고 영속성 컨텍스트를 다루는 창구이다.
7. 영속성 컨텍스트는 엔티티를 관리하는 메모리 공간이다.
8. 엔티티 상태는 비영속, 영속, 준영속, 삭제로 나뉜다.
9. `persist()`는 비영속 객체를 영속 상태로 만드는 것이다.
10. 영속 상태 엔티티는 변경 감지를 통해 수정 SQL 없이도 값 변경이 반영될 수 있다.
11. `detach()`는 특정 엔티티를 준영속 상태로 만들고, `merge()`는 다시 관리 흐름에 합치는 것이다.
12. `clear()`는 전체 준영속화, `close()`는 `EntityManager` 종료이다.
13. 트랜잭션은 JPA에서 변경 내용을 DB에 안전하게 확정하는 경계이다.
14. `given / when / then`은 테스트를 준비, 실행, 검증으로 나누는 읽기 좋은 구조이다.
15. `@ParameterizedTest`는 하나의 테스트를 여러 입력으로 반복 실행하게 해준다.
16. `@ValueSource`, `@CsvSource`, `@MethodSource`는 테스트 데이터를 공급하는 방식이 각각 다르다.
17. `@PersistenceContext`는 스프링 환경에서 `EntityManager`를 주입받는 방식이다.
