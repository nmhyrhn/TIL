# TIL - 장보기 연동과 사전 재료 관리자 기능 정리

장보기 화면을 실제 백엔드 API와 연결하고, 관리자용 사전 재료 관리 기능까지 함께 정리했다.  
기존 장보기 화면은 상태 중심으로만 동작했는데, 이번에는 추가, 조회, 체크, 삭제, 냉장고 반영까지 실제 API 흐름으로 바꾸면서 화면과 서버 데이터가 연결되도록 맞췄다.

특히 장보기 연동에서는 단순히 API를 붙이는 것보다, 항목 추가 후 목록 갱신, 체크 상태 변경, 삭제 후 반영, 냉장고 이동 이후 화면 반영처럼 사용자 흐름이 끊기지 않게 처리하는 점이 중요했다.  
장보기 기능은 단독 기능이라기보다 냉장고 기능으로 이어지는 중간 단계라는 점도 더 분명해졌다.

관리자 기능에서는 사전 재료 전체 조회, 비활성 목록 조회, 재료 추가, 수정, 비활성화, 재활성화 API를 구현했다.  
여기서 사용자용 API와 관리자용 API는 목적이 다르기 때문에 응답 구조도 분리하는 편이 자연스럽다는 점을 정리할 수 있었다.

또한 사전 재료 삭제를 실제 삭제 대신 `isActive` 기반 비활성화 방식으로 관리하면서, 데이터를 없애는 것과 상태값으로 관리하는 방식의 차이도 다시 확인했다.  
재료명 중복은 DB 에러에 맡기지 않고 서비스에서 먼저 검증해 예외를 명확히 처리하도록 한 점도 의미 있었다.

## 핵심 정리

- 장보기 화면을 실제 API 기반으로 연동했다.
- 장보기 기능에서는 API 호출 자체보다 화면 상태 갱신 흐름이 더 중요했다.
- 관리자용 사전 재료 API는 사용자용과 다르게 전체/비활성 데이터까지 관리해야 한다.
- 사전 재료는 하드 삭제보다 비활성화 방식이 더 적절했다.
- 중복 검증, 권한 구분, 응답 DTO 분리가 점점 더 중요해지는 구간이었다.
