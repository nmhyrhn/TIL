# TIL - JPA 매핑, 내장 타입, 연관관계, 그리고 언제 `@Transactional`을 쓰는가

## 1. 학습 주제

이번에는 JPA에서 엔티티를 어떻게 매핑하는지, 값 타입을 어떻게 재사용하는지, 엔티티끼리 어떤 식으로 연결되는지까지 범위가 넓어졌다.  
이전까지는 `엔티티 하나를 저장하고 조회하는 흐름`에 집중했다면, 이번에는 그 엔티티 안의 구조를 더 객체지향적으로 만드는 방법을 배우는 느낌에 가까웠다.

핵심 흐름은 다음과 같았다.

- 엔티티 안에 공통 값 묶음을 넣는 내장 타입
- 엔티티와 엔티티를 연결하는 연관관계 매핑
- `ManyToOne`, `OneToMany` 차이
- 즉시 로딩과 지연 로딩
- 서비스 계층에서 `@Transactional`이 왜 필요한지
- DTO와 엔티티의 역할 차이
- JPA 테스트를 읽는 데 필요한 `@ParameterizedTest`, `@MethodSource`, `@Autowired`

---

## 2. 매핑이란 무엇인가

JPA에서 매핑은 결국 이 뜻이다.

**자바 객체 구조를 데이터베이스 구조와 연결하는 규칙**

예를 들면 다음을 정하는 것이다.

- 이 클래스는 어떤 테이블과 연결되는가
- 이 필드는 어떤 컬럼과 연결되는가
- 이 enum은 문자열로 저장할지 숫자로 저장할지
- 이 필드는 DB에 저장하지 않을지
- 다른 엔티티와는 어떤 관계인지

즉 매핑은 “JPA가 이 객체를 DB에 어떻게 읽고 저장할지 알려주는 설명서”라고 보면 된다.

---

## 3. `@Embedded`와 `@Embeddable`

이 부분은 처음 보면 상속처럼 느껴질 수 있는데, 상속과는 다르다.

### `@Embeddable`
다른 엔티티 안에 **값 묶음으로 포함될 수 있는 클래스**라는 뜻이다.

### `@Embedded`
그 `@Embeddable` 클래스를 **내 엔티티 안에 내장해서 쓰겠다**는 뜻이다.

즉 관계를 정리하면:

- `@Embeddable` = 내장될 수 있는 값 타입 선언
- `@Embedded` = 그 값을 실제 엔티티 안에 넣어서 사용

---

## 4. `@Embedded`는 상속이 아니라 조합이다

이게 제일 중요하다.

`@Embedded`는 상속이 아니다.  
상속은 “부모 클래스의 성질을 자식이 물려받는 것”이고,  
`@Embedded`는 “한 객체 안에 다른 값 객체를 넣는 것”이다.

즉 느낌은 이런 쪽에 가깝다.

- 상속: “is-a” 관계
- `@Embedded`: “has-a” 관계

예를 들어 책 엔티티 안에 가격 정보가 들어간다고 하면:

- 책은 가격이다 → 이상함
- 책은 가격 정보를 가진다 → 자연스러움

그래서 `@Embedded`는  
**상속이 아니라 값 객체를 조합해서 재사용하는 방식**이라고 이해해야 한다.

---

## 5. `@Embeddable`의 목적


### 목적
**공통적으로 함께 쓰이는 값을 하나의 객체로 묶어서 재사용하기 위해서**

예를 들어 가격 정보를 생각해보면:

- 정가
- 할인율
- 판매가

이 세 개는 서로 관계가 있는 값들이다.  
이걸 엔티티에 낱개 필드로 다 흩뿌려 두는 것보다,  
`Price` 같은 값 객체로 묶으면 의미가 더 분명해진다.

즉 `@Embeddable`은  
**의미 있는 값 묶음을 객체로 표현하게 해주는 장치**다.

---

## 6. 왜 내장 타입을 쓰는가

장점은 생각보다 크다.

### 1. 의미가 분명해진다
그냥 `regularPrice`, `discountRate`, `sellPrice`를 흩뿌리는 것보다  
`price`라는 이름 아래 묶여 있으면 읽기가 좋다.

### 2. 재사용 가능하다
가격 구조가 여러 엔티티에서 반복되면 같은 값 타입을 재사용할 수 있다.

### 3. 검증 로직을 한 곳에 모을 수 있다
예제에서도 가격 객체 안에서:

- 음수 가격 방지
- 음수 할인율 방지
- 판매가 계산

을 처리하고 있었다.

즉 내장 타입은  
**단순 필드 묶음이 아니라, 값과 그 값을 다루는 규칙을 같이 묶는 방식**으로 이해하면 좋다.

---

## 7. `@Embedded`가 붙은 클래스는 실제로 어떻게 저장되나

이것도 많이 헷갈린다.

예를 들어 엔티티 안에 `@Embedded private Price price;`가 있다고 해서  
DB에 `price`라는 별도 테이블이 생기는 건 아니다.

오히려 반대다.

- 자바 코드에서는 `Price` 객체 하나처럼 보이지만
- DB에서는 그 안의 필드들이 현재 엔티티 테이블 컬럼으로 풀려서 저장된다

즉 내장 타입은  
**자바 코드에서는 객체로 묶고, DB에서는 현재 테이블 컬럼으로 저장되는 구조**다.

그래서 `@Embedded`는 “테이블 분리”가 아니라  
“객체 구조 정리”에 더 가깝다.

---

## 8. 지금 하는 작업은 DML인가

이 질문도 중요하다.

### DML
Data Manipulation Language  
즉 데이터를 조작하는 작업이다.

대표적으로:

- INSERT
- UPDATE
- DELETE
- SELECT

가 있다.

JPA에서:

- `persist()` -> INSERT 계열
- 엔티티 값 변경 후 commit -> UPDATE 계열
- `remove()` -> DELETE 계열
- `find()` -> SELECT 계열

로 연결된다.

즉 지금 JPA에서 하고 있는 저장, 수정, 삭제, 조회는 전부 넓게 보면 DML 작업이라고 볼 수 있다.  
다만 JPA는 SQL을 직접 쓰지 않고 객체 조작으로 표현할 뿐이다.

---

## 9. DTO와 엔티티 차이

이건 정말 중요하다.

### 엔티티
- JPA가 관리하는 객체
- 테이블과 매핑됨
- 영속성 컨텍스트의 관리 대상
- 저장/수정/삭제 흐름과 직접 연결됨

### DTO
- 데이터 전달용 객체
- JPA가 관리하지 않음
- 보통 요청/응답, 계층 간 데이터 이동에 사용
- 영속성 컨텍스트와 무관

즉 엔티티는 “DB 저장 대상”,  
DTO는 “데이터 운반 상자”라고 생각하면 된다.

### 왜 둘을 나누나
바로 엔티티를 외부 요청에 노출하면:

- 내부 구조가 그대로 드러나고
- 불필요한 값까지 열릴 수 있고
- 검증 책임도 애매해진다

그래서 보통은:

- 요청 받을 때 DTO
- 내부에서 엔티티로 변환
- 저장 처리

흐름을 많이 쓴다.

## 10. `@Autowired`

`@Autowired`는 Spring이 필요한 객체를 자동으로 넣어주는 기능이다.

예를 들어 테스트에서 서비스가 필요하면  
직접 `new Service()` 하지 않고 Spring이 Bean을 찾아 넣어준다.

즉:

- 개발자가 객체를 직접 조립하지 않고
- Spring 컨테이너가 의존 객체를 연결해준다

는 의미다.

테스트 코드에서 서비스에 `@Autowired`가 붙어 있으면  
“테스트에서도 Spring 컨테이너가 진짜 Bean을 주입해주는구나” 정도로 이해하면 된다.

---

## 11. `@Test`와 `@ParameterizedTest` 차이

### `@Test`
테스트 메서드를 **한 번 실행**한다.

입력이 하나거나, 한 가지 시나리오만 확인할 때 쓴다.

### `@ParameterizedTest`
같은 테스트 메서드를 **여러 입력값으로 반복 실행**한다.

즉 테스트 로직은 같고, 데이터만 여러 개 바꿔가며 확인하고 싶을 때 사용한다.

예를 들어 회원 등록 테스트를:

- 일반 회원 데이터
- 관리자 데이터

로 각각 돌리고 싶다면 `@ParameterizedTest`가 편하다.

즉 차이는 이것이다.

- `@Test` = 한 번 실행
- `@ParameterizedTest` = 같은 로직을 여러 데이터로 반복 실행

---

## 12. `@MethodSource`

`@MethodSource`는 `@ParameterizedTest`에 데이터를 공급하는 방식 중 하나다.

즉 테스트에 들어갈 값들을 메서드가 만들어서 넘겨준다.

왜 쓰냐면:

- 입력값이 여러 개일 때
- 문자열 하나가 아니라 복잡한 데이터 조합일 때
- 한 테스트를 다양한 케이스로 반복하고 싶을 때

유용하기 때문이다.

정리하면:

- `@ParameterizedTest` = 반복 테스트 실행 장치
- `@MethodSource` = 그 반복에 넣을 테스트 데이터를 공급하는 방식

이다.

---

## 13. Association Mapping이란 무엇인가

### Association Mapping
**연관관계 매핑**

즉 엔티티와 엔티티 사이의 관계를 JPA에서 객체 참조 형태로 표현하는 것이다.

예를 들어 메뉴와 카테고리를 생각하면:

DB에서는 보통:

- 메뉴 테이블
- 카테고리 테이블
- 외래키

구조로 표현된다.

객체에서는:

- 메뉴 객체 안에 카테고리 객체 참조
- 카테고리 객체 안에 메뉴 리스트 참조

같이 표현할 수 있다.

즉 연관관계 매핑은  
**외래키 중심 DB 관계를, 객체 참조 관계로 바꾸는 것**이라고 보면 된다.

---

## 14. `ManyToOne`과 `OneToMany` 차이

이건 꼭 방향부터 잡아야 한다.

### `ManyToOne`
“많은 쪽에서 하나를 참조”

예:
- 메뉴 여러 개
- 카테고리 하나

즉 메뉴 입장에서 보면  
“나는 하나의 카테고리를 가진다”이다.

그래서 메뉴 엔티티에서 카테고리를 들고 있는 구조가 `ManyToOne`이다.

### `OneToMany`
“하나가 많은 쪽을 참조”

예:
- 카테고리 하나
- 메뉴 여러 개

즉 카테고리 입장에서 보면  
“나는 여러 메뉴를 가진다”이다.

그래서 카테고리 엔티티에서 메뉴 리스트를 들고 있는 구조가 `OneToMany`이다.

### 한 줄 감각
- `ManyToOne` = 자식 -> 부모
- `OneToMany` = 부모 -> 자식들

---

## 15. 보통 `ManyToOne`이 더 자연스러운 이유

실무에서는 보통 `ManyToOne` 쪽이 더 자연스럽게 느껴진다.

왜냐하면 DB 외래키도 보통 “많은 쪽” 테이블에 있기 때문이다.

예:
- `tbl_menu.category_code`

처럼 메뉴 테이블이 카테고리 FK를 가진다.

그래서 객체에서도 메뉴가 카테고리를 참조하는 형태가  
DB 구조와도 더 가까워서 다루기 편하다.

반면 `OneToMany`는 컬렉션을 다뤄야 하므로  
조회 성능, 로딩 범위, 관리 복잡도가 더 커질 수 있다.

---

## 16. 지연 로딩은 왜 하는가

이 질문이 정말 중요하다.

### 지연 로딩
연관된 객체를 **당장 바로 조회하지 않고, 정말 필요할 때 조회하는 방식**

왜 필요하냐면, 연관 객체를 매번 즉시 다 가져오면 비효율적이기 때문이다.

예를 들어 메뉴 하나를 조회했는데,  
항상 카테고리 전체 정보까지 무조건 바로 끌어오면:

- 필요 없는 SQL이 실행될 수 있고
- 성능이 나빠질 수 있고
- 객체 그래프가 커질수록 더 무거워진다

그래서 “일단 메뉴만 가져오고, 카테고리가 정말 필요할 때 그때 읽자”는 전략이 지연 로딩이다.

즉 지연 로딩의 목적은  
**불필요한 연관 객체 조회를 줄여서 성능과 자원 사용을 최적화하는 것**이다.

---

## 17. 왜 `ManyToOne`에 `LAZY`를 붙이나

`ManyToOne`은 기본이 즉시 로딩인 경우가 많다.  
그런데 실무에서는 지연 로딩으로 바꾸는 경우가 많다.

이유는 간단하다.

메뉴를 조회할 때마다 항상 카테고리가 꼭 필요한 건 아니기 때문이다.

예:
- 메뉴 코드, 이름, 가격만 보여주는 화면
- 카테고리 정보는 나중에 필요할 수도 있음

이때 즉시 로딩이면 매번 같이 조회한다.  
지연 로딩이면 필요할 때만 가져온다.

그래서 보통 연관관계는 기본적으로 LAZY로 생각하는 경우가 많다.

---

## 18. `OneToMany`는 왜 더 조심해야 하나

`OneToMany`는 컬렉션을 들고 있으므로,  
즉시 로딩을 해버리면 한 번에 너무 많은 데이터가 딸려올 수 있다.

예를 들어 카테고리 하나를 조회했는데 그 안의 메뉴 목록까지 전부 같이 끌고 오면:

- 데이터 양이 많아지고
- 쿼리가 무거워지고
- 의도치 않은 성능 문제가 생길 수 있다

그래서 컬렉션 쪽은 특히 더 로딩 전략을 신중히 봐야 한다.

즉 `OneToMany`는 “편해 보이지만 무거워질 수 있다”는 감각을 가지는 게 좋다.

---

## 19. `cascade = PERSIST`는 무슨 뜻인가

이것도 함께 이해해야 한다.

### cascade
연관된 엔티티에 작업을 전파하는 것

예를 들어 메뉴를 저장할 때 연관된 카테고리도 같이 저장되게 하고 싶다면  
`CascadeType.PERSIST`를 줄 수 있다.

즉:

- 메뉴를 `persist()`
- 연결된 카테고리도 함께 `persist()`

되도록 전파하는 것이다.

중요한 건 cascade는 편리하지만 조심해서 써야 한다는 점이다.  
무심코 걸면 원하지 않는 저장/삭제까지 함께 일어날 수 있다.

---

## 20. `@Transactional`은 언제 써야 하나

트랜잭션은 언제 쓰는지가 중요하다.

### 기본 원칙
**DB 상태를 바꾸는 작업에는 거의 필수**

예:
- 등록
- 수정
- 삭제

이건 당연히 트랜잭션이 필요하다.

### 조회인데도 필요한 경우
조회 메서드에도 경우에 따라 트랜잭션을 쓸 수 있다.

대표적으로:

- 지연 로딩된 연관 객체를 서비스 메서드 안에서 안전하게 사용해야 할 때
- 같은 영속성 컨텍스트 안에서 읽기 작업을 유지해야 할 때

즉 쓰기 작업은 거의 무조건,  
읽기 작업은 영속성 컨텍스트/지연 로딩이 얽히면 필요할 수 있다.

### 한 줄 기준
- 저장/수정/삭제: 쓴다
- 단순 조회: 보통 안 붙여도 되지만 상황 따라 붙는다
- 지연 로딩을 같은 흐름 안에서 다뤄야 하면 붙이는 경우가 있다

---

## 21. 이번 흐름에서 `@Transactional`이 중요한 이유

서비스 메서드에서 엔티티를 저장하거나,  
연관관계가 있는 엔티티를 함께 다루거나,  
지연 로딩 객체를 접근하는 순간

단순 메서드 실행이 아니라:

- 영속성 컨텍스트 유지
- 엔티티 상태 추적
- flush/commit 시점 관리

가 다 연결된다.

즉 `@Transactional`은 단순히 commit/rollback만을 위한 게 아니라,  
**JPA가 엔티티를 안정적으로 관리하고 반영할 수 있게 해주는 실행 범위**라고 이해하면 더 정확하다.

---

## 22. 내장 타입과 연관관계를 함께 보면 좋은 점

이번 범위를 한 번에 보면, JPA가 객체지향적으로 설계하려는 방향이 더 잘 보인다.

### 내장 타입
가격처럼 함께 움직이는 값들을 하나의 값 객체로 묶는다.

### 연관관계 매핑
외래키 중심의 테이블 관계를 객체 참조 관계로 바꾼다.

즉 JPA는 단순히 “SQL 대신 메서드 호출”이 아니라,

- 값은 값답게 묶고
- 관계는 객체 참조로 표현하고
- 저장은 JPA가 맡게 하는

방향으로 설계를 유도한다.

---

## 23. 학습 흐름 정리

### 내장 타입
- `@Embeddable`로 값 객체 정의
- `@Embedded`로 엔티티 안에 내장
- 상속이 아니라 조합
- 공통 값 묶음 재사용과 검증 로직 집중

### 연관관계 매핑
- `ManyToOne`: 많은 쪽이 하나를 참조
- `OneToMany`: 하나가 여러 개를 참조
- 외래키 관계를 객체 참조 관계로 표현
- 로딩 전략과 cascade까지 함께 고려

### 테스트와 스프링
- `@Autowired`로 Bean 주입
- `@Test`는 한 번 실행
- `@ParameterizedTest`는 여러 입력으로 반복 실행
- `@MethodSource`는 복잡한 테스트 데이터 공급
- `@Transactional`은 쓰기 작업과 지연 로딩 처리에 중요

---

## 24. 핵심 정리

1. 매핑은 자바 객체 구조와 DB 구조를 연결하는 규칙이다.
2. `@Embeddable`은 내장 가능한 값 타입을 정의하고, `@Embedded`는 그것을 엔티티 안에 포함시킨다.
3. `@Embedded`는 상속이 아니라 조합이다.
4. 내장 타입은 공통 값 묶음을 재사용하고, 검증 로직을 한 곳에 모으는 데 유리하다.
5. 엔티티는 JPA가 관리하는 객체이고, DTO는 데이터 전달용 객체이다.
6. Association Mapping은 엔티티 간 관계를 객체 참조로 표현하는 것이다.
7. `ManyToOne`은 많은 쪽이 하나를 참조하고, `OneToMany`는 하나가 여러 개를 참조한다.
8. 지연 로딩은 연관 객체를 필요할 때만 조회해서 불필요한 SQL과 성능 낭비를 줄이기 위한 전략이다.
9. `cascade = PERSIST`는 연관된 엔티티에도 저장 작업을 전파하는 설정이다.
10. `@Transactional`은 저장, 수정, 삭제 같은 쓰기 작업에 거의 필수이며, 지연 로딩을 안전하게 다루는 데도 중요하다.
11. `@Test`는 단일 실행, `@ParameterizedTest`는 같은 테스트를 여러 입력으로 반복 실행하는 방식이다.
12. `@MethodSource`는 파라미터 테스트에 들어갈 복잡한 테스트 데이터를 메서드로 공급하는 방식이다.
13. `@Autowired`는 Spring이 필요한 객체를 자동 주입하게 하는 기능이다.

JPA는 문법 하나하나보다 “왜 객체를 이렇게 묶고, 왜 관계를 참조로 두고, 왜 로딩 전략을 나누는지”를 이해할수록 훨씬 쉬워진다.  
복습할 때는 `내장 타입은 값 객체`, `연관관계는 객체 참조`, `트랜잭션은 JPA 관리 범위` 이 세 문장을 중심으로 보면 도움이 많이 된다.
