# TIL - 장보기 도메인 API 구현과 냉장고 반영 흐름 정리

## 1. 학습 주제

장보기 도메인 API를 구현하면서, 단순 CRUD를 넘어서 사용자 기준 데이터 제어와 도메인 간 흐름 연결을 함께 정리했다.  
특히 장보기 항목을 추가하고 조회하는 기능뿐 아니라, 구매 여부를 관리하고 냉장고 재료로 반영하는 흐름까지 이어지는 구조를 구현했다.

- 장보기 항목 추가 API
- 장보기 목록 조회 API
- 장보기 항목 삭제 API
- 장보기 항목 체크/체크 해제 API
- 장보기 항목 냉장고 반영 API
- 로그인 사용자 기준 데이터 제어
- 장보기 도메인과 냉장고 도메인의 연결

---

## 2. 장보기 도메인의 핵심 흐름

장보기 기능은 단순히 “살 물건 메모”가 아니라, 이후 냉장고 재료 관리와 이어지는 중간 단계 역할을 한다.

전체 흐름은 다음처럼 볼 수 있다.

```text
재료를 사기 위해 장보기 항목 추가
→ 장보기 목록에서 확인
→ 구매 여부 체크
→ 구매 완료된 항목을 냉장고 재료로 반영
→ 장보기 목록에서는 제거
```

즉 장보기는 독립적인 기능이면서도, 냉장고 기능으로 이어지는 연결 지점이다.

---

## 3. 장보기 항목 추가 API

장보기 항목 추가 API는 사용자가 필요한 재료를 장보기 목록에 등록하는 기능이다.

요청에서는 사전 재료 ID와 수량을 받는다.

```json
{
  "productId": 1,
  "quantity": "1개"
}
```

여기서 중요한 점은 `memberId`를 요청으로 받지 않는다는 것이다.  
로그인한 사용자의 정보는 토큰에서 가져온 email을 기준으로 회원을 조회해서 처리한다.

이 방식이 중요한 이유는 다음과 같다.

- 요청값으로 회원 ID를 받으면 다른 사용자의 데이터를 건드릴 위험이 있다.
- 인증된 사용자 정보를 서버가 직접 가져와야 안전하다.
- 클라이언트는 “누가 요청했는지”가 아니라 “무엇을 추가할지”만 보내면 된다.

또한 요청으로 들어온 `productId`가 실제 존재하는 사전 재료인지 먼저 검증했다.  
존재하지 않는 재료를 장보기 항목으로 등록하면 이후 목록 조회나 냉장고 반영 과정에서 문제가 생길 수 있기 때문이다.

초기 저장 시 구매 여부는 아직 완료되지 않은 상태이므로 `isPurchased = false`로 저장했다.

## 4. 장보기 목록 조회 API

장보기 목록 조회 API는 로그인한 사용자의 장보기 항목만 조회하는 기능이다.

조회 조건은 다음 기준으로 잡았다.

```text
member_id = 로그인한 회원 id
is_deleted = false
```

즉,

- 내 데이터만 조회
- 삭제되지 않은 항목만 조회

하도록 구성했다.

여기서 중요한 점은 장보기 테이블에 재료 이름이나 카테고리 같은 정보가 직접 다 들어있는 것이 아니라는 점이다.  
장보기 항목에는 `productId`만 있고, 실제 재료명이나 카테고리 정보는 별도의 재료 테이블에 있다.

그래서 목록 응답을 만들 때는

```text
ShoppingItem + Product = 응답 DTO
```

형태로 합쳐서 반환했다.

이 과정에서 항목마다 상품을 하나씩 조회하지 않고, 먼저 `productId` 목록을 모은 뒤 한 번에 관련 정보를 가져오도록 구성했다.  
이 방식은 조회 횟수를 줄일 수 있어서 훨씬 효율적이다.

즉, 목록 조회는 단순히 장보기 테이블만 읽는 것이 아니라  
**장보기 데이터와 재료 데이터를 조합해서 사용자에게 필요한 형태로 반환하는 작업**이었다.

---

## 5. 장보기 항목 삭제 API

장보기 항목 삭제는 하드 삭제가 아니라 소프트 삭제 방식으로 구현했다.

즉, DB row를 실제로 없애는 것이 아니라 삭제 상태만 변경한다.

예를 들면 다음과 같은 방식이다.

```java
public void delete() {
    this.isDeleted = true;
    this.deletedAt = LocalDate.now();
}
```

이렇게 하면 실제 데이터는 남아 있고, 목록 조회 시 `isDeleted = false` 조건으로 걸러서 보이지 않게 만들 수 있다.

소프트 삭제를 쓰는 이유는 보통 다음과 같다.

- 사용자 실수 복구 가능성
- 삭제 이력 관리
- 운영 중 데이터 추적 필요성

또한 삭제할 때도 단순히 `shoppingItemId`만 보는 것이 아니라,

```text
shopping_item_id = 요청한 항목 id
member_id = 로그인한 회원 id
is_deleted = false
```

조건으로 조회해서 본인 항목만 삭제할 수 있게 했다.

이 부분이 중요하다.  
삭제 API에서 회원 조건이 빠지면 다른 사용자의 데이터를 삭제할 수 있는 심각한 문제가 생길 수 있기 때문이다.

---

## 6. 장보기 항목 체크/체크 해제 API

이 기능은 사용자가 해당 항목을 이미 구매했는지 표시하는 역할을 한다.

```text
false = 아직 구매하지 않음
true = 구매 완료
```

사용자는 요청값으로 `isPurchased`를 보내고, 서버는 그 값에 따라 상태를 변경한다.

이 기능은 얼핏 단순해 보이지만, 실제 사용자 경험에서는 꽤 중요하다.

- 아직 사지 않은 항목과 이미 산 항목을 구분할 수 있다.
- 냉장고 반영 전에 구매 완료 여부를 관리할 수 있다.
- 장보기 목록을 체크리스트처럼 사용할 수 있다.

즉, 장보기는 단순 목록이 아니라 상태를 가진 목록이라는 점이 드러난다.

## 7. 장보기 항목 냉장고 반영 API

이번 구현에서 가장 의미 있는 기능은 장보기 항목을 냉장고 재료로 반영하는 API였다.

이 기능은 단순 조회나 수정이 아니라,  
**한 도메인의 데이터를 다른 도메인으로 이동시키는 흐름**이다.

장보기 항목에는 이미 다음 정보가 들어 있다.

- productId
- quantity

그래서 냉장고 반영 요청에서는 중복으로 이 값을 다시 받을 필요가 없다.  
대신 냉장고 저장에 추가로 필요한 값만 받는다.

예를 들면 다음과 같은 정보다.

```json
{
  "expiryDate": "2026-07-05",
  "memo": "장보기에서 냉장고로 추가"
}
```

처리 흐름은 다음과 같다.

```text
1. 로그인 회원 조회
2. 내 장보기 항목 단건 조회
3. 장보기 항목의 productId, quantity 사용
4. 냉장고 재료 생성
5. 냉장고 테이블 저장
6. 기존 장보기 항목 삭제 처리
7. 응답 반환
```

이 흐름의 핵심은 다음 한 줄로 정리할 수 있다.

```text
장보기 항목 → 냉장고 재료
```

즉, 장보기는 끝나는 지점이 아니라 냉장고 등록으로 이어지는 출발점 역할도 한다.

---

## 8. 왜 `@Transactional`이 필요한가

냉장고 반영 기능에서는 `@Transactional`이 특히 중요하다.

이유는 이 API가 한 번에 두 가지 작업을 하기 때문이다.

1. 냉장고 재료 저장
2. 장보기 항목 삭제 처리

만약 트랜잭션이 없다면 다음 같은 문제가 생길 수 있다.

- 냉장고 저장은 성공했는데 장보기 삭제는 실패
- 장보기 삭제는 성공했는데 냉장고 저장은 실패

이러면 데이터가 꼬이게 된다.

예를 들어 첫 번째 경우에는 같은 항목이 장보기에도 남아 있고 냉장고에도 생겨버린다.  
두 번째 경우에는 장보기 목록에서는 사라졌는데 냉장고에는 추가되지 않는 문제가 생긴다.

그래서 이 두 작업은 반드시 **한 덩어리로 성공하거나, 한 덩어리로 실패해야 한다.**  
이럴 때 `@Transactional`을 사용한다.

즉, 트랜잭션은 “메서드에 붙이는 기술”이 아니라  
**데이터 정합성을 지키기 위해 여러 DB 작업을 하나로 묶는 장치**다.

## 9. 로그인한 사용자 기준 처리의 중요성

이번 장보기 API 구현 전체에서 가장 중요한 기준 중 하나는  
**모든 데이터가 로그인한 사용자 기준으로 처리되어야 한다는 점**이었다.

공통적으로 적용된 흐름은 다음과 같다.

- 토큰에서 사용자 식별 정보 추출
- 해당 사용자 조회
- 이후 모든 장보기 항목 조회/수정/삭제 시 회원 조건 추가

즉, path variable이나 request body만 믿고 처리하는 것이 아니라  
항상 “이 요청을 보낸 사람이 누구인지”를 먼저 확인한 뒤 본인 데이터만 다루도록 한 것이다.

이 원칙은 장보기뿐 아니라 거의 모든 사용자 개인 데이터 기능에서 매우 중요하다.

예를 들면

- 내 장바구니
- 내 주문
- 내 냉장고
- 내 메모

같은 기능도 모두 같은 기준으로 봐야 한다.

## 10. 계층별 역할 정리

이번 구현을 통해 다시 정리된 계층별 역할은 다음과 같다.

### Controller
- 요청값 받기
- 인증 사용자 정보 받기
- 서비스 호출
- 응답 반환

### Service
- 회원 검증
- 본인 데이터 조회
- 비즈니스 로직 처리
- 다른 도메인과 연결되는 흐름 제어

### Repository
- DB 조회 및 저장

### Entity
- 자기 상태 변경
- 예: 삭제 처리, 구매 상태 변경

### DTO
- 요청/응답 구조 분리
- 외부와 주고받는 데이터 형태 관리

이 구조를 따라가면 코드가 길어져도 각 클래스의 책임이 비교적 분명해진다.

## 11. 구현하면서 정리된 포인트

1. `memberId`를 요청으로 받지 않고 인증 정보에서 가져오는 방식이 더 안전하다.
2. 목록 조회는 한 테이블만 보는 것이 아니라 다른 도메인 데이터를 조합할 수 있다.
3. 삭제는 하드 삭제보다 소프트 삭제가 더 적절한 경우가 있다.
4. 체크/체크 해제 같은 단순 상태 변경도 사용자 흐름에서는 중요한 기능이다.
5. 장보기와 냉장고는 분리된 도메인이지만 실제 사용 흐름에서는 자연스럽게 연결된다.
6. 여러 DB 작업이 한 번에 일어나는 기능은 트랜잭션으로 묶어야 한다.
7. 사용자 데이터 기능에서는 항상 “본인 데이터만 처리하는가”를 먼저 확인해야 한다.

---

## 12. 핵심 정리

장보기 도메인은 단순 CRUD로 끝나는 기능이 아니라,  
구매 전 목록 관리와 구매 후 냉장고 반영까지 이어지는 흐름 중심 도메인으로 이해할 수 있었다.

특히 이번 구현에서는 다음이 핵심이었다.

- 로그인한 사용자 기준 데이터 제어
- 장보기 항목 상태 관리
- 장보기와 냉장고 도메인 연결
- 트랜잭션을 통한 데이터 일관성 보장

장보기 기능은 앞으로 프론트와 연결될 때도 흐름이 중요한 기능이다.  
사용자가 항목을 추가하고, 체크하고, 냉장고로 옮기는 과정이 자연스럽게 이어져야 하기 때문에 백엔드에서도 그 흐름을 끊기지 않게 설계하는 것이 중요하다는 점을 정리할 수 있었다.
