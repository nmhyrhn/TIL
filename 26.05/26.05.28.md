# TIL - Spring Boot 요청 매핑, 핸들러 메서드, 세션 처리, 그리고 AOP

## 1. 학습 주제


1. **Spring Boot에서 HTTP 요청이 컨트롤러 메서드로 어떻게 연결되는지**
2. **AOP가 서비스 로직에 어떻게 공통 기능을 끼워 넣는지**

이번 범위는 Spring MVC의 진입점에 가까운 내용이라,  
“브라우저 요청이 서버 코드 어디로 들어가고, 어떤 규칙으로 메서드가 선택되는지”를 이해하는 게 핵심이었다.

---

## 2. Spring Boot에서 요청을 받는 방식

이전까지는 서블릿에서 직접:

- URL 확인
- HTTP 메서드 확인
- 분기 처리
- request/response 직접 읽기

를 많이 했다면, Spring Boot에서는 그 과정을 훨씬 선언적으로 바꿔준다.

즉 개발자는:

- “이 URL은 이 메서드가 처리해”
- “이 쿼리스트링은 이 파라미터로 받아”
- “이 JSON 본문은 DTO로 바꿔줘”

라고 선언만 하면, Spring이 중간 작업을 대신 처리해준다.

이게 Spring MVC가 편한 가장 큰 이유다.

---

## 3. `@RestController`와 `@Controller`

이번 예제는 대부분 JSON 응답을 다뤘기 때문에 `@RestController`를 사용했다.

### `@Controller`
- 주로 HTML 화면 반환
- 뷰 이름 반환에 자주 사용

### `@RestController`
- 주로 JSON 응답
- `@Controller + @ResponseBody` 성격

즉 API 서버처럼 동작하는 코드를 만들 때는 `@RestController`를 많이 쓴다.

이번 학습의 컨트롤러들은 대부분 “화면 이동”보다 “JSON 데이터 반환”이 목적이었기 때문에 `@RestController`가 자연스러운 선택이었다.

---

## 4. 클래스 단위 요청 매핑

컨트롤러 클래스 위에 `@RequestMapping`을 붙이면,  
그 클래스 안의 모든 메서드에 공통 URL prefix를 줄 수 있다.

예제에서는 이런 구조였다.

- 주문 관련 요청은 `/api/v1/orders`
- 메뉴 관련 요청은 `/api/v1/menus`

이 방식의 장점은:

- URL 구조를 보기 좋게 정리할 수 있고
- 비슷한 기능끼리 한 컨트롤러로 묶기 쉽고
- 메서드마다 prefix를 반복해서 쓰지 않아도 된다

즉 클래스 단위 매핑은 “큰 주소 틀”을 잡는 역할이다.

---

## 5. 메서드 단위 요청 매핑

클래스 안에서는 각 메서드가 HTTP 메서드와 세부 경로를 담당한다.

예를 들어:

- `@GetMapping` : 조회
- `@PostMapping` : 등록
- `@PutMapping` : 수정
- `@DeleteMapping` : 삭제

이렇게 나눌 수 있다.

즉 같은 `/menus`라는 큰 주소 안에서도:

- GET `/menus`
- POST `/menus`
- PUT `/menus/{id}`
- DELETE `/menus/{id}`

처럼 HTTP 메서드에 따라 전혀 다른 동작을 하게 만들 수 있다.

이게 REST 스타일 API의 기본 감각이다.

---

## 6. `ResponseEntity`가 하는 일

예제에서는 응답을 `ResponseEntity`로 많이 감쌌다.

이 객체는 단순히 “데이터만” 보내는 게 아니라 다음을 함께 담을 수 있다.

- 응답 본문(body)
- HTTP 상태 코드(status)
- 필요하면 헤더(header)

예를 들면:

- 정상 조회: `200 OK`
- 생성 성공: `201 Created`

처럼 응답 의미를 더 정확하게 표현할 수 있다.

즉 `ResponseEntity`는  
“응답 데이터 + 상태 코드”를 함께 다루는 도구라고 이해하면 된다.

---

## 7. 요청 매핑 챕터의 핵심 흐름

요청 매핑 예제는 결국 이런 질문에 답하는 내용이었다.

- 어떤 URL이 어떤 컨트롤러로 갈까?
- 같은 URL이어도 GET, POST가 다르면 다른 메서드로 갈까?
- 경로 변수는 어떻게 받을까?

예제 기준 흐름은 이렇다.

### 메뉴 목록 조회
- GET `/api/v1/menus`
- 메뉴 목록 조회용 메서드 호출

### 메뉴 등록
- POST `/api/v1/menus`
- 새 메뉴 등록용 메서드 호출

### 메뉴 수정
- PUT `/api/v1/menus/3`
- `3`번 메뉴 수정 메서드 호출

### 메뉴 삭제
- DELETE `/api/v1/menus/2`
- `2`번 메뉴 삭제 메서드 호출

즉 Spring은 URL과 HTTP 메서드를 보고  
적절한 컨트롤러 메서드를 자동으로 선택해준다.

---

## 8. 핸들러 메서드란 무엇인가

핸들러 메서드는  
**실제로 요청을 받아 처리하는 컨트롤러 안의 메서드**이다.

예를 들어:

- 메뉴 목록 조회 메서드
- 메뉴 등록 메서드
- 세션 저장 메서드
- 헤더 확인 메서드

같은 것들이 전부 핸들러 메서드다.

핵심은 Spring이 이 메서드들을 호출할 때,  
매개변수도 같이 해석해서 넣어준다는 점이다.

즉 예전처럼 `request.getParameter()`를 직접 다 하지 않아도,  
Spring이 매개변수에 맞게 값을 바인딩해준다.

---

## 9. `@RequestParam`

`@RequestParam`은 쿼리스트링 값을 받을 때 사용한다.

예:
- `GET /api/v1/menus?keyword=coffee&categoryCode=2&page=1`

이 요청에서:

- `keyword`
- `categoryCode`
- `page`

값을 메서드 파라미터로 받을 수 있다.

중요한 점:

- `required = false`를 주면 없어도 된다
- `defaultValue`를 주면 값이 없을 때 기본값을 사용할 수 있다

즉 `@RequestParam`은  
“URL 뒤에 붙은 `?key=value` 형식의 값을 자바 변수로 바꾸는 어노테이션”이다.

### 전체 Map으로 받기
쿼리스트링이 많거나 유동적일 때는 `Map<String, String>`으로 한 번에 받을 수도 있다.

이건 “검색 조건이 다양할 때” 꽤 편리한 방식이다.

---

## 10. `@PathVariable`

`@PathVariable`은 URL 경로 일부를 변수처럼 받는 방식이다.

예:
- GET `/api/v1/menus/3`

여기서 `3`은 경로의 일부다.  
이 값을 메서드 파라미터로 받는 것이 `@PathVariable`이다.

즉:

- `/menus/{menuCode}`
- `{menuCode}` 자리에 들어온 값을
- 자바 변수 `menuCode`로 받는 구조

이건 보통 “특정 자원 하나를 조회/수정/삭제”할 때 많이 쓴다.

---

## 11. `@RequestBody`

`@RequestBody`는 요청 본문(body)에 들어 있는 JSON을 자바 객체로 바꿔준다.

예를 들어 클라이언트가 이런 JSON을 보내면:

```json
{
  "name": "아이스 아메리카노",
  "price": 5000,
  "categoryCode": 2,
  "orderableStatus": "Y"
}
```

Spring은 이걸 DTO 객체로 변환해서 메서드 파라미터에 넣어준다.

즉 `@RequestBody`는  
“JSON 요청 본문 -> 자바 객체” 변환을 담당한다.

이게 중요한 이유는,  
현대 백엔드 개발에서 POST/PUT 요청은 대부분 JSON으로 데이터를 받기 때문이다.

---

## 12. DTO가 왜 필요한가

JSON 본문을 그냥 문자열로만 받으면 불편하다.  
그래서 DTO를 만든다.

DTO는:

- 클라이언트가 보낸 데이터를 한 객체로 묶고
- 서버 안에서 다루기 쉽게 해준다

예를 들면 메뉴 등록 요청이라면:

- 이름
- 가격
- 카테고리 코드
- 판매 여부

를 하나의 DTO 객체로 받는 게 훨씬 깔끔하다.

즉 DTO는 “요청 데이터 꾸러미” 역할을 한다.

---

## 13. `@RequestBody`를 문자열로 받는 경우

예제에서는 JSON이 아니라 raw text도 받아봤다.

이 경우 `@RequestBody String body`처럼 문자열로 바로 받을 수 있다.

즉 `@RequestBody`는 꼭 DTO만 되는 게 아니라:

- JSON이면 DTO
- 그냥 plain text면 String

처럼 사용할 수도 있다.

이건 요청 본문 자체를 확인하거나,  
특수한 포맷을 직접 처리하고 싶을 때 유용하다.

---

## 14. `@RequestHeader`

HTTP 요청에는 body뿐 아니라 header도 있다.

예를 들어:

- `User-Agent`
- `Content-Type`
- `Authorization`
- `Accept`

같은 정보들이다.

`@RequestHeader`를 사용하면 이 값을 바로 받을 수 있다.

예:
- 특정 헤더 하나만 받기
- 전체 헤더를 Map으로 받기

즉 `@RequestHeader`는  
“요청 메타데이터를 읽는 도구”라고 보면 된다.

실무에서는 특히 인증 토큰, 브라우저 정보, content type 확인 등에 자주 사용된다.

---

## 15. `@CookieValue`

쿠키 값도 바로 받을 수 있다.

예제에서는 `JSESSIONID` 같은 쿠키 값을 읽는 흐름이 있었다.

이걸 통해 알 수 있는 점은:

- 쿠키도 결국 HTTP 요청에 실려서 오고
- Spring이 이를 어노테이션으로 쉽게 꺼내주며
- 직접 `request.getCookies()`를 반복해서 파싱하지 않아도 된다는 점이다

즉 `@CookieValue`는  
“요청에 포함된 쿠키 값을 바로 바인딩해주는 방식”이다.

---

## 16. `HttpSession` 사용

세션도 핸들러 메서드에서 직접 받을 수 있었다.

예제에서는:

- 세션에 값 저장
- 세션에서 값 조회
- 세션 만료

를 다뤘다.

이 흐름은 로그인 상태 관리에 매우 중요하다.

### 저장
POST 요청으로 아이디 같은 값을 세션에 넣는다.

### 조회
GET 요청으로 세션 안에 저장된 값을 다시 확인한다.

### 만료
DELETE 요청 등으로 세션을 무효화한다.

즉 Spring MVC에서는 세션도 컨트롤러 매개변수로 자연스럽게 다룰 수 있다.

---

## 17. 요청 처리 방식 전체 흐름

이번 학습 내용을 한 번 전체로 묶어 보면, Spring Boot는 요청을 이렇게 처리한다.

1. 클라이언트가 URL로 요청
2. Spring이 URL과 HTTP 메서드를 보고 적절한 컨트롤러 메서드 선택
3. 요청 데이터 분석
4. 쿼리스트링은 `@RequestParam`
5. 경로 값은 `@PathVariable`
6. JSON body는 `@RequestBody`
7. 헤더는 `@RequestHeader`
8. 쿠키는 `@CookieValue`
9. 세션은 `HttpSession`
10. 컨트롤러가 응답 데이터 생성
11. `ResponseEntity`로 상태 코드와 함께 반환

즉 “요청을 어떻게 받는지”를 매우 세밀하게 다룬 파트였다.

## 18. AOP 다시 연결해서 보기

같은 날 AOP 예제도 조금 더 보강되었다.

AOP는  
**공통 관심사를 핵심 로직에서 분리해, 필요한 지점에 자동으로 끼워 넣는 방식**이었다.

이번 예제에서는 `LoggingAspect`가 서비스 메서드 실행 전에 로그를 찍는다.

즉 서비스 메서드 안에 로그 코드를 직접 쓰지 않고도:

- 어떤 서비스 객체인지
- 어떤 메서드가 실행되는지

를 출력할 수 있다.

이게 중요한 이유는:

- 로그
- 트랜잭션
- 보안 검사
- 실행 시간 측정

같은 기능이 여러 서비스에 반복되기 때문이다.

즉 요청 처리 흐름과 연결해서 보면:

- 컨트롤러가 서비스 호출
- 서비스 메서드가 실행되기 직전
- AOP가 먼저 끼어들어 공통 기능 실행
- 그 뒤 핵심 로직 실행

이런 구조가 된다.

---

## 19. AOP의 핵심 구성 감각

### Target
실제 핵심 로직 객체  
예: 서비스 객체

### Aspect
공통 기능을 담는 클래스  
예: 로그 출력 클래스

### Pointcut
어디에 적용할지 조건  
예: 이름이 `Service`로 끝나는 클래스의 메서드

### Advice
언제 실행할지  
예: `@Before` -> 메서드 실행 전

즉 AOP는 “공통 기능을 따로 빼고, 적용 지점을 정해서 자동으로 실행하는 구조”라고 보면 된다.

## 20. 이번 학습에서 특히 중요한 포인트

### 1. Spring Boot는 URL을 메서드에 자연스럽게 연결해준다
예전보다 훨씬 선언적이다.

### 2. 컨트롤러 메서드 파라미터가 매우 강력하다
요청 데이터를 종류별로 다르게 받을 수 있다.

### 3. JSON API 만들 때 DTO와 `@RequestBody`는 핵심이다
실무에서 매우 자주 쓴다.

### 4. 세션/쿠키/헤더도 컨트롤러에서 쉽게 다룰 수 있다
HTTP 자체를 더 자연스럽게 다룰 수 있게 해준다.

### 5. AOP는 핵심 로직을 건드리지 않고 공통 기능을 적용한다
로그, 트랜잭션 같은 반복 코드를 분리하는 데 강력하다.

## 21. 흐름 정리

### 요청 매핑
- 클래스 단위 URL prefix 설정
- 메서드 단위로 GET/POST/PUT/DELETE 분기
- `ResponseEntity`로 응답 상태 코드와 body 함께 처리

### 핸들러 메서드
- `@RequestParam`으로 쿼리스트링 받기
- `@PathVariable`로 경로 값 받기
- `@RequestBody`로 JSON 요청 받기
- `@RequestHeader`, `@CookieValue`, `HttpSession`으로 요청 메타데이터 다루기

### AOP
- 공통 관심사와 핵심 관심사 분리
- Pointcut으로 대상 선택
- `@Before`로 실행 전 공통 기능 적용
- 서비스 로직 바깥에서 로깅 수행

---

## 22. 핵심 정리

1. Spring Boot에서는 URL과 HTTP 메서드를 기준으로 컨트롤러 메서드가 자동 매핑된다.
2. `@RestController`는 JSON 응답 중심 API를 만들 때 자주 사용한다.
3. 클래스 단위 `@RequestMapping`은 공통 URL prefix를 잡는 역할을 한다.
4. `@GetMapping`, `@PostMapping`, `@PutMapping`, `@DeleteMapping`은 HTTP 메서드별 요청을 처리한다.
5. `ResponseEntity`를 사용하면 body뿐 아니라 상태 코드까지 함께 제어할 수 있다.
6. `@RequestParam`은 쿼리스트링, `@PathVariable`은 URL 경로 값을 받을 때 사용한다.
7. `@RequestBody`는 JSON 요청 본문을 DTO로 바인딩하는 데 핵심적이다.
8. `@RequestHeader`, `@CookieValue`, `HttpSession`을 통해 요청의 부가 정보와 상태도 다룰 수 있다.
9. AOP는 로그, 트랜잭션처럼 반복되는 공통 기능을 핵심 로직 밖으로 분리하는 방식이다.
10. `@Aspect`, Pointcut, Advice를 조합하면 서비스 메서드 실행 전후에 공통 기능을 자동 적용할 수 있다.
