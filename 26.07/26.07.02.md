# TIL - 냉장고 유통기한 자동 반영 보완

냉장고와 장보기 흐름에서 사전 재료의 기본 유통기한 값이 실제 사용자 흐름까지 이어지도록 보완한 작업이었다.

기존에는 사전 재료에 `defaultExpiryDays` 값이 있어도, 장보기 항목을 냉장고로 반영할 때 유통기한이 비어 저장될 수 있었다. 
또한 관리자 사전재료 등록 화면에서도 기본 유통기한을 입력할 수 없어 데이터 관리가 불편했다.

## 진행 내용

- 장보기 항목을 냉장고로 반영할 때 기본 유통기한 자동 적용
- 관리자 사전재료 등록/수정 폼에 기본 유통기한 입력 연결
- `defaultExpiryDays`가 DB, API, 화면 흐름에서 끊기지 않도록 보완
- 프론트 빌드, 백엔드 컴파일, 관련 테스트 확인

## 장보기 항목 냉장고 반영 시 기본 유통기한 적용

장보기 항목을 냉장고로 옮길 때 사용자가 유통기한을 직접 입력하지 않으면, 사전 재료의 기본 유통기한을 기준으로 날짜를 자동 계산하도록 수정했다.

처리 기준은 다음과 같다.

| 상황 | 처리 방식 |
|---|---|
| 요청에 `expiryDate`가 있는 경우 | 요청 날짜 우선 사용 |
| 요청에 `expiryDate`가 없고 `defaultExpiryDays`가 있는 경우 | 현재 날짜 + 기본 유통기한 일수 |
| `defaultExpiryDays`가 없는 경우 | 기존 흐름 유지 |

핵심은 요청값을 먼저 존중하고, 비어 있을 때만 사전 재료의 기본값을 사용하는 것이다.

```java
LocalDate expiryDate = request.expiryDate();

if (expiryDate == null && product.getDefaultExpiryDays() != null) {
    expiryDate = LocalDate.now().plusDays(product.getDefaultExpiryDays());
}
```

이후 냉장고 재료를 생성할 때는 `request.expiryDate()`를 그대로 넘기지 않고, 계산된 `expiryDate`를 사용하도록 했다.

```java
FridgeItem fridgeItem = FridgeItem.create(
        member.getId(),
        shoppingItem.getProductId(),
        shoppingItem.getQuantity(),
        expiryDate,
        request.memo()
);
```

이렇게 하면 사용자가 직접 날짜를 입력한 경우에는 해당 날짜가 저장되고, 입력하지 않은 경우에는 사전 재료의 기본 유통기한이 자동으로 적용된다.

## 관리자 사전재료 기본 유통기한 입력 연결

관리자 사전재료 관리 화면에서는 재료명과 카테고리는 입력할 수 있었지만, 기본 유통기한을 입력하는 칸이 없었다.

그래서 등록과 수정 화면에 숫자 입력칸을 추가하고, 요청 시 `defaultExpiryDays`에 포함되도록 연결했다.

```jsx
<input
  type="number"
  min="0"
  placeholder="유통기한(일)"
  value={addDefaultExpiryDays}
  onChange={(e) => setAddDefaultExpiryDays(e.target.value)}
/>
```

등록 요청에서는 입력값을 숫자로 변환해서 전달했다.

```javascript
adminApi.createProduct({
  productCategoryId: CATEGORY_IDS[addCategory],
  name: addName.trim(),
  defaultExpiryDays: addDefaultExpiryDays ? Number(addDefaultExpiryDays) : null,
});
```

수정 요청도 동일하게 `defaultExpiryDays`를 포함하도록 처리했다.

```javascript
adminApi.updateProduct(items[idx].productId, {
  productCategoryId: CATEGORY_IDS[editCategory],
  name: editName.trim(),
  defaultExpiryDays: editDefaultExpiryDays ? Number(editDefaultExpiryDays) : null,
});
```

목록에서는 기본 유통기한 값이 있으면 `7일`처럼 보여주고, 값이 없으면 `미설정`으로 표시했다.

## 백엔드 확인

관리자 사전재료 쪽 백엔드는 이미 `defaultExpiryDays` 흐름이 연결되어 있었다.

확인한 부분은 다음과 같다.

- 생성 요청 DTO에 `defaultExpiryDays` 존재
- 수정 요청 DTO에 `defaultExpiryDays` 존재
- 생성 서비스에서 `request.defaultExpiryDays()` 사용
- 수정 서비스에서 `request.defaultExpiryDays()` 사용
- 엔티티 생성/수정 시 `defaultExpiryDays` 저장
- 응답 DTO에 `defaultExpiryDays` 포함

따라서 관리자 사전재료 작업은 백엔드 추가 수정 없이, 프론트에서 입력값을 요청에 포함시키는 방식으로 처리했다.

## 검증

이번 작업에서 확인한 내용은 다음과 같다.

- 장보기 항목 냉장고 반영 테스트 통과
- 프론트 빌드 통과
- 백엔드 컴파일 통과
- 관리자 사전재료 등록 폼 입력칸 확인
- 관리자 사전재료 수정 폼 입력칸 확인
- `defaultExpiryDays` 요청 포함 확인

## 정리

이번 작업으로 사전 재료의 기본 유통기한 값이 실제 사용자 흐름에 더 자연스럽게 연결되었다.

관리자는 사전재료를 등록하거나 수정할 때 기본 유통기한을 함께 관리할 수 있고, 사용자는 장보기 항목을 냉장고로 옮길 때 유통기한을 직접 입력하지 않아도 기본값을 자동 적용받을 수 있다.

결과적으로 `DB 사전 재료 데이터 → 관리자 화면 → 장보기 → 냉장고 반영` 흐름이 하나의 기준으로 이어지게 되었다.
