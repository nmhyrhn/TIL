# TIL - 냉장고/장보기 QA와 사전 재료 데이터 정리

이번 작업은 냉장고/장보기 핵심 기능 QA와 사전 재료 데이터 정리, 그리고 화면 사용성 보완을 중심으로 진행했다.

냉장고와 장보기 API는 대부분 구현되어 있었기 때문에, 개별 API 구현보다 실제 사용자가 화면에서 기능을 사용하는 흐름이 자연스럽게 이어지는지 확인하는 데 더 집중했다.

## 진행 내용

- 냉장고/장보기 사용자 흐름 QA
- 사전 재료 DDL 정리
- 냉장고 유통기한 자동 반영 보완
- 장보기 즉시 구매 버튼 추가

## 냉장고/장보기 사용자 흐름 QA

QA에서는 단순히 API 응답 성공 여부만 보는 것이 아니라, 사용자가 실제로 화면에서 기능을 사용하는 순서대로 흐름을 점검했다.

냉장고에서는 다음 흐름을 확인했다.

- 사전 재료 검색
- 카테고리 목록 조회
- 냉장고 재료 등록
- 냉장고 재료 목록 조회
- 카테고리별 조회
- 냉장고 재료 수정
- 냉장고 재료 삭제
- 전부 사용 처리
- 일부 사용 처리
- 유통기한 임박 조회
- 만료 재료 조회

장보기에서는 다음 흐름을 확인했다.

- 장보기 항목 추가
- 장보기 목록 조회
- 장보기 항목 삭제
- 체크/체크 해제
- 냉장고 반영

핵심 사용자 흐름에서는 별도 버그로 분리할 정도의 문제는 발견되지 않았다.

## 사전 재료 DDL 정리

사전 재료 데이터를 보강하기 위해 products 테이블용 DDL을 다시 정리했다.

처음에는 많은 수의 데이터를 추가하려고 했지만, 색상, 국가명, 등급, 용량 등을 붙인 조합형 데이터는 실제 재료 데이터로 보기 어렵다고 판단했다.  
그래서 서비스에서 실제로 검색하고 등록할 만한 재료명 위주로 기준을 다시 잡았다.

정리 기준은 다음과 같았다.

- 한국어 재료명만 사용
- 실제 냉장고/장보기에서 사용할 만한 재료명만 반영
- 기존 이름과 중복 제거
- 카테고리와 기본 유통기한 함께 반영
- 개수를 억지로 늘리지 않음

최종적으로는 기존 재료에 신규 재료를 추가해 더 실사용에 가까운 사전 재료 구성을 만들 수 있었다.

## DDL 적용 중 확인한 점

신규 재료 INSERT 시 `product_id`를 직접 넣으면 기본키 충돌이 발생할 수 있다는 점도 정리했다.

`product_id`는 자동 생성 컬럼이므로 직접 넣지 않고, DB가 시퀀스로 생성하도록 처리하는 방식이 맞았다.  
수정 후에는 `product_id`를 제외하고 INSERT 하도록 바꾸고, 마지막에 시퀀스 보정까지 함께 처리했다.

이 과정에서 단순히 데이터를 넣는 것보다, 테이블 구조와 시퀀스 동작 방식까지 이해하고 반영하는 것이 중요하다는 점을 다시 확인했다.

## 냉장고 유통기한 자동 반영 보완

사전 재료에 `defaultExpiryDays` 값은 이미 있었지만, 냉장고 화면에서 재료를 선택했을 때 유통기한이 자동으로 입력되지 않는 문제가 있었다.

원인은 DB나 백엔드가 아니라 프론트 연결 누락이었다.  
백엔드 응답에는 기본 유통기한 값이 포함되어 있었지만, 프론트에서 화면용 데이터로 변환하는 과정에서 해당 값을 넘기지 않고 있었다.

이 부분을 보완하면서,

- 검색 결과에 `defaultExpiryDays`를 포함하고
- 재료 선택 시 현재 날짜에 해당 일수를 더해
- `expiryDate`를 자동 입력

하도록 연결했다.

이제 사용자는 재료를 선택하면 기본 유통기한이 자동으로 들어오고, 필요하면 직접 수정할 수도 있다.

## 장보기 즉시 구매 버튼 추가

장보기 항목에서 바로 검색으로 이어질 수 있도록 즉시 구매 버튼도 추가했다.

이번 범위에서는 별도 백엔드 API나 파트너스 연동까지 확장하지 않고, 프론트에서 재료명을 쿠팡 검색 URL로 변환해 새 탭으로 여는 방식으로 처리했다.

이 방식은 구조가 단순하고, 현재 단계에서 필요한 사용자 흐름을 빠르게 만들기에 충분했다.

즉,

- 장보기 항목 확인
- 버튼 클릭
- 바로 검색 이동

으로 이어지는 최소 기능을 먼저 완성한 셈이다.

## 테스트 및 확인

이번에 확인한 내용은 다음과 같다.

- 냉장고/장보기 핵심 사용자 흐름 정상 동작
- 신규 한국어 재료 데이터 반영
- 재료명 중복 없음
- 재료 선택 시 기본 유통기한 자동 입력
- 장보기 즉시 구매 버튼 동작 확인
- 기존 냉장고/장보기 기능 영향 없음

## 정리

이번 작업을 통해 API가 구현되어 있는 것과, 실제 사용자 입장에서 기능이 완성되어 보이는 것은 다를 수 있다는 점을 다시 느꼈다.

특히 `defaultExpiryDays`처럼 DB에도 있고 API 응답에도 있는 값이 프론트 연결 하나 빠졌다는 이유로 실제 기능이 없는 것처럼 보일 수 있었다.  
결국 기능은 한 군데만 완성된다고 끝나는 것이 아니라, DB부터 API, 프론트 상태, 화면 입력까지 끝까지 이어져야 사용자 입장에서 비로소 동작한다고 느낄 수 있다는 점을 정리할 수 있었다.
